Systems note · agent orchestration

The agent pipeline is a spiking neural network.

Every specialist agent is a neuron. It only runs — and only spends tokens — when its input crosses threshold. The agents that aren't needed never fire, so they cost nothing. Cost control isn't a rule bolted on top; it falls out of the dynamics.

01The core idea

A conventional agent pipeline is a fixed line: clarify → implement → review → log, every stage every time. This one is a network. A task is scored into "sensory current," that current spikes through a graph of LIF (leaky integrate-and-fire) neurons, and which neurons fire decides which agents run.

Because a neuron below threshold never fires, its agent never executes. A one-line rename lights up three neurons; a sprawling refactor lights the whole board. The cheap tasks stay cheap automatically — that's the whole point.

02The neurons

Two layers. Sensory neurons encode features of the incoming task; specialist neurons are the agents themselves, each with an action that runs a Claude Code call when it spikes.

03How a task flows

  1. Score. score_task() turns the task text into a 0–100 current per sensory neuron (a transparent heuristic — the one hand-tuned part).
  2. Stimulate. The sensory layer is driven with those currents. Any sensory neuron over threshold spikes and propagates along its synapses (a spike adds weight × 50 to each downstream neuron).
  3. Cascade. Spikes ripple through the graph. A specialist that crosses threshold fires — and its handler runs the agent synchronously, before the spike propagates onward. So the pipeline self-sequences: the Implementer's agent finishes, then its spike drives the Reviewer.
  4. Route on results. Some routing depends on what an agent found: if the Reviewer's output reports issues, it stimulates the Corrector — which re-drives the Reviewer once. Refractory periods stop that loop running away.
  5. Settle & log. When the network goes quiet, the ledger is written once. Neurons that never crossed threshold never ran, and cost nothing.

Two mechanisms keep it honest, both borrowed straight from spiking dynamics:

Inhibition — a real veto

The synapse S_Ambiguous → Implementer has weight −2.0. A strongly ambiguous task hyperpolarizes the Implementer — pushes it below rest — so even an exciting "there's work to do" signal can't lift it over threshold. You don't implement a task you don't understand, and that gate now lives in the network, not in a Python if. Verified: with ambiguity present and Work firing, the Implementer fires zero times; without it, it fires normally.

Refractory — no runaway

After a neuron fires it can't re-fire for a set window. That's what stops a correct → review → correct → … spiral from looping forever. The same mechanism that keeps a biological network from seizing keeps the pipeline from burning tokens in a circle.

05Why it's trustworthy

The pipeline sits on a discipline, not just a topology: pre-register a falsifiable claim before acting, then verify against it. Nothing is judged "done" by the agent that did it — an independent read-only reviewer checks the work, and a hard gate (compile, tests, one behavior assertion) decides pass or fail. That verification layer is what makes agent output safe to trust rather than merely fast. In practice it has caught a real defect on essentially every non-trivial change.

06Concurrency scheduling — three honest benchmarks

The same inhibitory-synapse mechanism was extended to a different problem: when many build-agents run at once, two editing the same file clobber each other. Model each agent as a neuron and every pair with an overlapping file-set as a mutual inhibitory synapse — one network "settle" reads off a conflict-free set of agents that may run concurrently. That's a genuinely different mechanism from a lock or a DAG. Whether it's a genuinely better one had to be measured, not assumed — so it was, three separate times, against the actual classical baseline each time.

Bottom line

The concurrency scheduler is not a differentiated algorithm and shouldn't be pitched as one externally — that specific claim is now measured false. It remains a correct, convenient internal implementation detail of this pipeline (zero conflicts across 1,700+ tested graphs). Every graded-conflict variant tested loses to plain uniform weighting; none of the three benchmarks found a practical win.

A retraction, shown rather than hidden

An earlier version of this table reported the ternary gate winning graded conflicts by ~4%. That number came from a benchmark whose "reproducible seed" wasn't actually reproducible — it seeded per-pair randomness with Python's built-in hash(), which is randomized per process by default. Fixed with a stable hash and re-run across 5 independent seeds: the result reverses. This is included because a page that only shows the results that survived isn't more honest than one that hides its failures — it's the same failure, later.
